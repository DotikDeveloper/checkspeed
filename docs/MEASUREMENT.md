# Методика измерения скорости

CheckSpeed реализует модель, близкую к [RMBT](https://www.netztest.at/doc/) и Ookla Speedtest: отдельные фазы для ping и throughput, несколько параллельных TCP-потоков внутри одного направления, расчёт скорости только по времени передачи полезной нагрузки.

## Единицы

- **Мбит/с** в интерфейсе — мебибиты в секунду: `(байты × 8) / (1024² × секунды)`.
- Размеры файлов — **мебибайты** (1024² байт).

## Общий порядок теста

```text
1. Ping (8 попыток, один раз за сессию UI)
2. Для каждой из 10 итераций UI:
   a. Download (получение)
   b. Upload (отдача)
```

Направления **не выполняются одновременно**, чтобы не делить полосу пропускания между download и upload.

Функция `testAllSpeeds()` выполняет: `ping` → `download` → `upload` последовательно.

## Ping (задержка)

| Параметр | Значение |
|----------|----------|
| Метод | `HEAD /api/ping` |
| Попыток | 8 |
| Интервал | 120 мс |
| Агрегация | минимум успешных замеров после обрезки краёв и IQR |

Замеры с нулевой задержкой (например, ответ **429**) в расчёт не попадают. Первая и последняя успешные попытки отбрасываются при ≥3 значениях (снижение эффекта «холодного старта» TCP).

## Download (получение)

### Параметры

| Параметр | Значение |
|----------|----------|
| Размеры | 2, 5, 8 МБ (последовательно) |
| Попыток на размер | 2 (+1 повтор при сбое) |
| Параллельных потоков | 3 (`PARALLEL_STREAMS`) |

На каждую попытку объём файла делится на **3 параллельных** `GET /api/download?size=…`. Скорость одной попытки:

```text
Mbps = (сумма байт всех потоков × 8) / (1024² × длительность_передачи_сек)
```

**Длительность передачи** — от первого байта payload (первый чанк любого потока) до последнего чанка последнего потока. **TTFB и установка соединения не входят** в знаменатель (`resolveThroughputDurationMs`).

### Агрегация

1. По размеру: удаление выбросов (IQR), среднее двух попыток.
2. Итог: взвешенное среднее по размерам (вес = размер в МБ).
3. Округление до целого Мбит/с для UI.

## Upload (отдача)

Логика зеркальна download:

- Payload создаётся в памяти (`Uint8Array`).
- Делится на 3 части, каждая отправляется отдельным `POST /api/upload` через `XMLHttpRequest`.
- Время передачи: `loadstart` → `loadend` на upload-канале XHR.
- Таймаут одного потока: **60 с**.

## Качество измерения в UI

Индекс **0–100** учитывает:

- долю успешных (не нулевых) замеров;
- коэффициент вариации download/upload;
- наличие валидного ping.

## Ограничения и точность

| Фактор | Влияние |
|--------|---------|
| Один сервер (same-origin) | Измеряется путь клиент ↔ ваш инстанс, не «весь интернет» |
| Serverless (Vercel) | Лимиты времени и CPU могут занижать скорость на очень быстрых каналах |
| 3 потока | Обычно достаточно до ~1 Гбит/с; для выше может потребоваться больше соединений |
| Фиксированный объём | На сверхбыстрых линиях короткое окно передачи даёт больший разброс |

## Публичные функции клиента

| Функция | Назначение |
|---------|------------|
| `testPing()` | Только ping |
| `testDownloadSpeed()` | Только download |
| `testUploadSpeed()` | Только upload |
| `testAllSpeeds()` | Полный цикл |
| `bytesToMbps(bytes, seconds)` | Конвертация |
| `aggregateParallelTransferSpeed(samples)` | Агрегация параллельных потоков |
| `findNearestServer(servers)` | Заготовка под multi-server |
